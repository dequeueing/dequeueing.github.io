This post walks through Exercise 1 of Fuzzing 101, where we use AFL++ to fuzz Xpdf and then use GDB to understand the root cause of a crash.

The goal is not only to run a fuzzer and wait for crashes. The more important part is to understand what AFL++ is doing, what the crash report means, and how to move from "the program crashed" to "this is the likely root cause."

## What We Are Testing {#what-we-are-testing}

The target program is `pdftotext`, a command-line tool from Xpdf. It takes a PDF file as input and converts it into text.

This is a good fuzzing target because PDF is a structured but complicated file format. A PDF parser needs to handle many object types, streams, dictionaries, xref tables, inline images, malformed files, and recovery paths. This complexity creates many opportunities for bugs.

In this exercise, AFL++ mutates PDF files and repeatedly runs `pdftotext` on those mutated inputs. If one input triggers a crash, AFL++ saves it so that we can reproduce and debug it.

## How AFL++ Works {#how-afl-works}

AFL++ is a **coverage-guided fuzzer**. This means it does not mutate inputs completely blindly. For each generated input, AFL++ observes which execution paths the target program takes.

The core feedback loop is:

```text
seed PDF files
   ->
AFL++ mutates them
   ->
pdftotext executes on the mutated PDFs
   ->
instrumentation records coverage
   ->
AFL++ keeps inputs that discover new paths
```

The key idea is simple: if a mutated input reaches a new execution path, it may expose code that previous inputs did not exercise. AFL++ keeps that input in its queue and uses it as a future mutation seed.

By default, AFL++ does not understand PDF semantics. It treats each PDF as a byte sequence and mutates bytes by flipping bits, inserting bytes, deleting chunks, changing integers, or combining mutations. Many generated inputs are invalid PDFs and will be rejected quickly by `pdftotext`. But some mutated inputs remain structured enough to drive the parser deeper, and these are the useful ones.

When source code is available, AFL++ can use **instrumentation**. In this exercise, Xpdf is compiled with an AFL++ compiler wrapper such as `afl-clang-fast`. The wrapper does not usually rewrite the source files directly. Instead, it inserts lightweight coverage-recording logic during compilation, producing an instrumented `pdftotext` binary.

## Running AFL++ {#running-afl}

The fuzzing command is:

```bash
afl-fuzz \
  -i $HOME/fuzzing_xpdf/pdf_examples/ \
  -o $HOME/fuzzing_xpdf/out/ \
  -s 123 \
  -- \
  $HOME/fuzzing_xpdf/install/bin/pdftotext \
  @@ \
  $HOME/fuzzing_xpdf/output
```

The important arguments are:

- `-i $HOME/fuzzing_xpdf/pdf_examples/`: the input seed corpus. This directory contains valid example PDFs.
- `-o $HOME/fuzzing_xpdf/out/`: the output directory where AFL++ stores its queue, crashes, hangs, and runtime state.
- `-s 123`: a fixed pseudo-random seed. AFL++ is non-deterministic, so setting a seed makes the run easier to reproduce.
- `--`: separates AFL++ options from the target program command.
- `@@`: placeholder that AFL++ replaces with the path of each generated input file.

So the real target program is:

```text
$HOME/fuzzing_xpdf/install/bin/pdftotext
```

At runtime, the process looks like this:

```text
Xpdf source code
   -> compiled with afl-clang-fast
instrumented pdftotext
   -> repeatedly executed by afl-fuzz
mutated PDF files
   -> fed into pdftotext
coverage feedback
   -> sent back to AFL++
interesting PDFs
   -> saved for further mutation
```

## Reading the Fuzzer Output {#reading-the-fuzzer-output}

After AFL++ starts running, the status screen tells us how the fuzzing campaign is progressing.

![Fuzzing result](/images/blog/fuzzing101-ex1/fuzzing_result.png)

Two useful fields are:

- `corpus count`: how many interesting inputs AFL++ has kept.
- `own finds`: how many of those inputs were discovered by AFL++ itself rather than coming from the initial seed corpus.

For example, `corpus count: 978` means AFL++ has kept many inputs that exercise different execution paths. `own finds: 977` means almost all of them were discovered through mutation rather than being part of the original seed set.

At this point, if there are no crashes or hangs, the right action is usually to keep the fuzzer running. AFL++ does not stop automatically when it feels "done"; we decide when to stop.

## Understanding the Crash Files {#understanding-the-crash-files}

Eventually, AFL++ found crashes.

![Crash information](/images/blog/fuzzing101-ex1/crash.png)

A crash filename contains useful metadata. For example:

```text
id:000000,sig:11,src:001186,time:300468,execs:635385,op:havoc,rep:1
```

This can be read as:

- `id:000000`: this is the first distinct crash sample saved by AFL++.
- `sig:11`: the target received signal 11, which is `SIGSEGV`. This usually means invalid memory access or stack overflow.
- `src:001186`: this crash was mutated from queue input `001186`.
- `time:300468`: the crash was found about 300 seconds after fuzzing started.
- `execs:635385`: AFL++ had executed the target about 635k times.
- `op:havoc`: the crash was found during AFL++'s havoc mutation stage.
- `rep:1`: the mutation operation was repeated once.

One important point: 17 crash files do not necessarily mean 17 different bugs. If they all have `sig:11` and many come from nearby source queue entries, they may be different inputs that trigger the same underlying bug.

## Reproducing the Crash {#reproducing-the-crash}

Before debugging, we should confirm that the crash can be reproduced outside AFL++.

First, inspect the crash file:

```bash
file "$CRASH"
```

The result shows that the input is still recognized as a PDF variant:

```text
/root/fuzzing_xpdf/out/default/crashes/id:000000,sig:11,src:001186,time:300468,execs:635385,op:havoc,rep:1: PDF document, version 1.7, 1 page(s)
```

Then run `pdftotext` manually:

```bash
"$HOME/fuzzing_xpdf/install/bin/pdftotext" \
  "$CRASH" \
  "$HOME/fuzzing_xpdf/crash-output"
```

The output ends with:

```text
Error: Bad image parameters
Segmentation fault
```

This confirms that the crash is reproducible without AFL++. That matters because a reproducible crash is much easier to debug and report.

The error messages before the segmentation fault are also useful. Many of them mention inline image dictionary issues, which gives us an early hint that the crash may happen while parsing an inline image.

## Debugging with GDB {#debugging-with-gdb}

Next, run the crashing input under GDB so that the program stops at the crash point.

![Run GDB](/images/blog/fuzzing101-ex1/run_gdb.png)

The backtrace shows the call chain:

![GDB backtrace](/images/blog/fuzzing101-ex1/trace.png)

The relevant stack is:

```text
pdftotext main
-> PDFDoc::displayPages
-> PDFDoc::displayPage
-> Page::display
-> Page::displaySlice
-> Gfx::display
-> Gfx::go
-> Gfx::opBeginImage
-> EmbedStream::getChar
-> crash
```

This is consistent with the earlier error messages. The crash likely happens while parsing or rendering an inline image.

At this point, the useful questions are:

- Which object is being accessed at the crash line?
- Is `this` a valid object?
- Are any internal member pointers null?
- Did an upper-layer function pass invalid state into this function?

Printing more frame-local information gives the key clue:

![Debug frame 1](/images/blog/fuzzing101-ex1/debug1.png)

![Debug frame 2](/images/blog/fuzzing101-ex1/debug2.png)

The important observation is:

```text
this->str = 0x00
```

So the crash is caused by dereferencing a null stream pointer.

## Root Cause {#root-cause}

Looking at the source code, `str` is assigned directly from `strA` without a null check. Further investigation shows that `parser->getStream()` returned null, and that null value was passed into the `EmbedStream` object.

So the problem is not simply that `EmbedStream::getChar` crashed. The deeper issue is that an invalid stream object was allowed to propagate into `EmbedStream`.

The fix is to add a null check around the stream returned by `parser->getStream()`, or otherwise ensure that `EmbedStream` is never constructed with a null stream.

This is the path from fuzzing to root cause:

```text
AFL++ finds crashing PDF
   ->
manual reproduction confirms SIGSEGV
   ->
GDB shows crash in EmbedStream::getChar
   ->
backtrace points to inline image parsing
   ->
object inspection shows this->str == null
   ->
source inspection shows missing null check after parser->getStream()
```

## Takeaways {#takeaways}

In this exercise, AFL++ found a crashing PDF input for Xpdf. The crash was reproducible outside AFL++, and GDB helped narrow it down to a null stream pointer during inline image parsing.

The broader lesson is that fuzzing is not only about generating crashes. A useful fuzzing workflow also includes:

1. understanding how the fuzzer generates and selects inputs;
2. reading the fuzzer's crash metadata;
3. reproducing the crash manually;
4. using a debugger to identify the failing object and call chain;
5. connecting the crash back to a concrete root cause in the source code.

That is the difference between "the fuzzer found something" and "we understand the bug."
